\subsection*{Glosario de términos}

\begin{itemize}
    \item \textbf{HU}: Historia de usuario. Descripción de una funcionalidad desde el punto de vista del usuario.
    \item \textbf{Alfajor}: Unidad de medida de complejidad usada por el equipo.
    \item \textbf{Task}: Tarea técnica concreta en la que se descompone una historia de usuario.
    \item \textbf{Definition of Done (DoD)}: Criterio que define cuándo una task se considera terminada.
    \item \textbf{Daily Scrum}: Reunión breve diaria del equipo para sincronizar avances y detectar impedimentos.
    \item \textbf{Task Board}: Tablero de tareas con columnas \textit{To Do}, \textit{In Progress} y \textit{Done}.
    \item \textbf{Story Point (SP)}: Unidad relativa de estimación de esfuerzo para las historias de usuario.
\end{itemize}

\subsection{Estándares de Desarrollo}

\subsubsection*{Repositorio del proyecto}

El proyecto se gestiona en un único repositorio grupal en GitHub, ya que el equipo desarrolla una sola aplicación tipo Classroom.

\textbf{Repositorio:} \url{https://github.com/Vargm-b/task-repository.git}

La estrategia de ramas definida para el Sprint 1 es la siguiente:

\begin{itemize}
    \item rama principal: \texttt{main};
    \item una rama individual por integrante con el formato \texttt{feature/nombre};
    \item cada integrante desarrolla sus tasks en su propia rama;
    \item los cambios se integran a \texttt{main} manteniendo la estructura común del sistema.
\end{itemize}

Esta organización permite trabajar en paralelo sin fragmentar la arquitectura del proyecto y facilita la integración al cierre del sprint.

\subsubsection*{Stack tecnológico}

\begin{itemize}
    \item \textbf{HTML5 + CSS3}: estructura y diseño visual de la interfaz.
    \item \textbf{JavaScript (ES6+)}: lógica del frontend.
    \item \textbf{Node.js}: entorno de ejecución del backend.
    \item \textbf{Express}: framework para la creación del servidor y definición de endpoints REST.
    \item \textbf{PostgreSQL en Supabase}: almacenamiento de datos en base compartida por el equipo.
\end{itemize}

\subsubsection*{Dependencias del backend}

\begin{itemize}
    \item \textbf{express} \texttt{5.1.0}: servidor HTTP y enrutamiento.
    \item \textbf{pg} \texttt{8.16.3}: cliente PostgreSQL para Node.js.
    \item \textbf{dotenv} \texttt{17.2.1}: lectura de variables de entorno desde \texttt{.env}.

\subsubsection*{Modelo de datos — Sprint 1}

Se definió la estructura de datos común en PostgreSQL a través de Supabase. Las tablas cubren el flujo funcional mínimo del Sprint 1: creación de clases, inscripción de estudiantes y publicación de tareas.

\begin{itemize}
    \item \textbf{Tabla \texttt{users}}
    \begin{itemize}
        \item \texttt{id}: identificador único del usuario (PK).
        \item \texttt{name}: nombre completo.
        \item \texttt{email}: correo electrónico único.
        \item \texttt{role}: tipo de usuario (\texttt{teacher} o \texttt{student}).
    \end{itemize}

    \item \textbf{Tabla \texttt{classes}}
    \begin{itemize}
        \item \texttt{id}: identificador único de la clase (PK).
        \item \texttt{name}: nombre de la clase virtual.
        \item \texttt{description}: descripción general.
        \item \texttt{access\_code}: código único de acceso generado por el sistema.
        \item \texttt{teacher\_id}: referencia al docente creador (\texttt{users.id}).
    \end{itemize}

    \item \textbf{Tabla \texttt{enrollments}}
    \begin{itemize}
        \item \texttt{id}: identificador único de la inscripción (PK).
        \item \texttt{class\_id}: referencia a la clase (\texttt{classes.id}).
        \item \texttt{student\_id}: referencia al estudiante inscrito (\texttt{users.id}).
    \end{itemize}

    \item \textbf{Tabla \texttt{assignments}}
    \begin{itemize}
        \item \texttt{id}: identificador único de la tarea (PK).
        \item \texttt{class\_id}: referencia a la clase (\texttt{classes.id}).
        \item \texttt{title}: título de la tarea.
        \item \texttt{description}: descripción o instrucciones.
        \item \texttt{max\_score}: puntaje máximo asignado.
        \item \texttt{due\_date}: fecha y hora límite de entrega.
        \item \texttt{attachment\_url}: URL del archivo adjunto (opcional).
        \item \texttt{created\_at}: fecha y hora de creación (autogenerado).
    \end{itemize}
\end{itemize}

\subsubsection*{Relaciones entre tablas}

\begin{itemize}
    \item \texttt{classes.teacher\_id} $\rightarrow$ \texttt{users.id}
    \item \texttt{enrollments.class\_id} $\rightarrow$ \texttt{classes.id}
    \item \texttt{enrollments.student\_id} $\rightarrow$ \texttt{users.id}
    \item \texttt{assignments.class\_id} $\rightarrow$ \texttt{classes.id}
\end{itemize}

\subsubsection*{Configuración de la conexión a base de datos}

Todos los integrantes utilizan las mismas variables de entorno para conectarse a la base de datos compartida en Supabase. El archivo \texttt{.env} debe contener:

\begin{verbatim}
DB_USER=valor
DB_HOST=valor
DB_NAME=valor
DB_PASSWORD=valor
DB_PORT=valor
\end{verbatim}

Este archivo \textbf{no debe subirse al repositorio}; debe incluirse en \texttt{.gitignore}.

    \item Instalar dependencias: \texttt{npm install}
    \item Crear el archivo \texttt{.env} con las credenciales de conexión.

\subsubsection*{Convención de nombres}

El equipo usa \textbf{inglés} para todos los identificadores técnicos (tablas, atributos, rutas, archivos, variables y funciones). La documentación se redacta en español.

\begin{itemize}
    \item Tablas y columnas: \texttt{snake\_case}, minúsculas, plural.
    \item Variables y funciones JS: \texttt{camelCase}.
    \item Archivos y carpetas: minúsculas, sin espacios, guión medio permitido.
    \item Consistencia de nombres entre frontend, backend y base de datos.
\end{itemize}

\subsubsection*{Convención de commits}

Los commits se escriben en inglés con mensajes breves y orientados a la acción, siguiendo el formato:

\begin{verbatim}
type(scope): short description
\end{verbatim}

Tipos definidos:

\begin{itemize}
    \item \texttt{feat}: nueva funcionalidad.
    \item \texttt{fix}: corrección de errores.
    \item \texttt{docs}: cambios en documentación.
    \item \texttt{refactor}: reorganización interna sin cambiar comportamiento.
    \item \texttt{style}: ajustes de formato o estilos visuales.
    \item \texttt{chore}: tareas técnicas menores o de mantenimiento.
\end{itemize}

Ejemplos válidos:
\begin{itemize}
    \item \texttt{feat(classes): add create class endpoint}
    \item \texttt{feat(enrollments): validate access code on join}
    \item \texttt{feat(assignments): add due date field to form}
    \item \texttt{fix(frontend): correct class join form validation}
    \item \texttt{docs(sprint0): update development standards}
\end{itemize}

\subsubsection*{Política de integración de cambios}

\begin{itemize}
    \item No trabajar directamente sobre \texttt{main}.
    \item Actualizar la rama personal antes de integrar (\texttt{git pull origin main}).
    \item Verificar localmente que la funcionalidad no rompe la estructura común.
    \item Comunicar al equipo cualquier conflicto detectado.
    \item No marcar una task como \textit{Done} sin evidencia funcional e integración completada.
\end{itemize}
